Mac配置iTerm2实现mosh自动远程连接

发表于 2018-08-25 更新于 2020-04-14 分类于 Linux ， 教程

其实网上教程也挺多的，但是用起来总是不那么方便，还遇到了一些bug。现在就总结几个问题再输出一下！

1 准备

首先创建一个ssh秘钥是最方便的，这样就不用输入密码，不过用了该功能也没必要了，直接用就是的。

1.1 安装expect

首先确保你的mac安装了expect，这个是一个自动化执行脚本，可以输入一些普通shell脚本做不到的东西，需要人工操作的。比如回车确认等等。

brew install expect

如果你brew都没装那可以出门右转了。

1.2 安装mosh

你也可以不装，只使用ssh连接，那么直接跳过这里。

这个是什么呢？就是用来解决ssh卡顿的，因为默认的是键盘每输入一个字符就要向服务器发送一个字节，这样就会大大增加传输次数，也会增加延迟。

mosh 是一个替代SSH的免费软件，它最大的特点是支持网络漫游和间歇性连接。

  • 会话的中断不会导致当前正在前端执行的命令中断，相当于你所有的操作都是在screen命令中一样在后台执行
  • 会话在中断过后，不会立刻退出，而是启用一个计时器，当网络恢复后会自动重新连接，同时会延续之前的会话，不会重新开启一个
brew install mosh

目前只是客户端mac安装了，服务器端也要安装。如果是Ubuntu的话用如下命令：

sudo apt-get install mosh

看到这里你可能觉得有点复杂，但是一劳永逸，亲测延迟要降低很多！

然后运行mosh-server

mosh-server

这个时候mosh-server可能绑定的是60001端口，可以自己看一下

netstat -anp | grep mosh

像我是阿里云的服务器，可能就要到安全组里开通60001的udp端口，具体怎么设置相信你可以百度得到。

如果提示错误：

那么可能是你的服务器没有设置中文的utf-8，执行以下命令：

sudo apt-get -y install language-pack-zh-hans

如果弹出

那么就意味着运行成功了。

2 配置

准备操作已完成，那么就进行配置了。

2.1 脚本

首先利用expect在/usr/local/bin/下面写一个脚本login.exp，并加上执行权限：

chmod +x login.exp

login.exp

使用mosh

#!/usr/bin/expectset timeout 30spawn mosh -p 60001 [lindex $argv 0]@[lindex $argv 1]expect {    "(yes/no)?"    {send "yes\n";exp_continue}    "password:"    {send "[lindex $argv 2]\n"}}interact

使用ssh

#!/usr/bin/expectset timeout 30spawn ssh [lindex $argv 0]@[lindex $argv 1]expect {    "(yes/no)?"    {send "yes\n";exp_continue}    "password:"    {send "[lindex $argv 2]\n"}}interact

2.2 iTerm2配置

可以直接按快捷键command + ,，或者在iTerm2设置里打开preference，点开Profiles，然后在Genernal->Command下选择 Login shell，在输入框里填入相关信息，用空格隔开，如果有特殊字符需要用单引号隔开。

你还可以自己加一个快捷键

3 使用

使用快捷键一键登录！享受mosh带来的低延迟吧！

硬链接和软连接的区别

发表于 2018-08-22 更新于 2020-04-14 分类于 Linux

  • .

    在Linux的文件系统中，保存在磁盘分区中的文件不管是什么类型都给它分配一个编号，称为索引节点号inode 。软连接，其实就是新建立一个文件，这个文件就是专门用来指向别的文件的（那就和windows 下的快捷方式的那个文件有很接近的意味）。软链接产生的是一个新的文件，但这个文件的作用就是专门指向某个文件的，删了这个软连接文件，那就等于不需要这个连接，和原来的存在的实体原文件没有任何关系，但删除原来的文件，则相应的软连接不可用（cat那个软链接文件，则提示“没有该文件或目录“）

    硬连接是不会建立inode的，他只是在文件原来的inode link count域再增加1而已，也因此硬链接是不可以跨越文件系统的。相反都是软连接会重新建立一个inode，当然inode的结构跟其他的不一样，他只是一个指明源文件的字符串信息。一旦删除源文件，那么软连接将变得毫无意义。而硬链接删除的时候，系统调用会检查inode link count的数值，如果他大于等于1，那么inode不会被回收。因此文件的内容不会被删除。

    硬链接实际上是为文件建一个别名，链接文件和原文件实际上是同一个文件。可以通过ls -i来查看一下，这两个文件的inode号是同一个，说明它们是同一个文件；而软链接建立的是一个指向，即链接文件内的内容是指向原文件的指针，它们是两个文件。

    软链接可以跨文件系统，硬链接不可以；软链接可以对一个不存在的文件名(filename)进行链接（当然此时如果你vi这个软链接文件，linux会自动新建一个文件名为filename的文件）,硬链接不可以（其文件必须存在，inode必须存在）；软链接可以对目录进行连接，硬链接不可以。两种链接都可以通过命令 ln 来创建。ln 默认创建的是硬链接。使用 -s 开关可以创建软链接。

Tcpdump和Wireshark抓包简单教程

总结一下抓包的过程，主要是一个熟悉流程的过程。

1 Linux抓包

抓包的一个传统流程，当然使用的是tcpdump。

1.1 定位

首先抓包不能一通乱抓，要确定你想知道哪个进程的网络数据交换过程，那么当然要知道进程ID和该进程所使用的端口。

主要是针对自己工作的SSLVPN抓包过程。

ps命令

ps -aux | grep 进程名

使用该命令获取进程的ID。

netstat命令

netstat -anp |grep 进程名

这也可以获得端口号和进程ID，直接省略了PS过程。

1.2 抓包

使用经典的tcpdump来抓包，建议保存到本地用wireshark看。

tcpdump -i eth0 port 8801 -c 100 -X -s 0 -w sgip.cap
//抓所有8801端口的数据包，其他的IP可以看网上教程

几个重要的选项：

  • -c，抓包的个数；
  • -X，显示包的内容；
  • -s 0，不截断，显示完整长度。

1.3 分析包（TCP包）

还有很多包，UDP、DNS等等。

主要是一些三次握手是否正常，断开是否异常等。 然后还可以看包体的数据： 这是前面4个是TCP/IP的4层模型，最后一个是包自己的协议数据。

突然想起以前自己做项目的时候，遇到网络问题没什么没有想起抓包呢？可能那个时候还不会吧，现在想来一抓包，问题都能分析出来了。

2 Window抓包

与Linux不同的是，Windows没有那么多便捷的命令可以直接查看数据。

2.1 定位

如果你想捕获某个程序的包，那么也需要知道其用的端口。可以在任务管理其查看。

  • 查看-选择列-勾选进程PID

  • 找到指定进程的PID

netstat命令

和Linux有些许不同。

netstat -ano | findstr 进程ID

这个时候就可以监控进程所使用的端口了。

2.2 抓包

这个时候可以直接使用wireshark进行抓包，只要在过滤条件下设置一下你要的端口或者IP就能显示你想要的数据了。

添加过滤条件显示

可以看一下这篇文章: wireshark过滤语法总结

分析过程就不再赘述了，可以参考这篇文章：WireShark抓包 图解探索网络请求过程（五层网络模型、三次握手、滑动窗口协议）

阿里云服务器Ubuntu下Apache2配置https

教程``posts``SSL

最近把网站搬迁了，然后HTTPS也得弄一弄，本文主要是介绍一下阿里云服务器如何给自己的网站配置https

1 购买证书

这个证书的下载比较鸡贼，阿里云把免费的给藏起来了，先点击这里： 然后再点击增强型，这个时候就会出现免费的证书选项了。 然后点击免费的购买

2 下载证书

将证书里的四个文件放在/etc/apache2/cert，自己建一个cert文件夹。

3 配置SSL

进入目录

cd /etc/apache2/sites-available

编辑default-ssl.conf文件，修改域名和目录

ServerAdmin webmaster@localhost
ServerName 你的域名:443
DocumentRoot /var/www/html/你的网站目录

然后在SSLEngine on的下面加入如下内容：

SSLProtocol all -SSLv2 -SSLv3
SSLCipherSuite HIGH:!RC4:!MD5:!aNULL:!eNULL:!NULL:!DH:!EDH:!EXP:+MEDIUM
SSLHonorCipherOrder on
SSLCertificateFile  /etc/apache2/cert/public.pem
SSLCertificateKeyFile /etc/apache2/cert/你的.key
SSLCertificateChainFile /etc/apache2/cert/chain.pem

4 配置HTTPS

编辑000-default.conf，在<VirtualHost *:80></VirtualHost>标签内随便一个地方加入以下三行:

RewriteEngine on
RewriteCond  %{HTTPS} !=on
RewriteRule  ^(.*) https://%{SERVER_NAME}$1 [L,R]

5 重启

记得载入模块：

a2ensite default-ssl
#然后重启apache2
sudo etc/init.d/apache2 restart

6 备注

记得目录一定要对，然后看一下阿里云服务器是否开放了443这个端口，配置一下安全组规则。 推荐一个wordpress插件：Really Simple SSL 但是如果有时候你关闭了SSL，要访问http形式的网站，但这个时候你进不了网站，关不了这个插件，可以参考这个强制关闭该插件的方法：

进入wordpress目录

cd wp-content/plugins/really-simple-ssl

修改文件

将改txt后缀改为php

mv force-deactivate.txt force-deactivate.php

用http访问该PHP

http://www.yourdomain.com/wp-content/plugins/really-simple-ssl/force-deactivate.php

这个时候就可以卸载该插件关闭全站https了。

Linux下编写小脚本提高效率

Linux下编写小脚本提高效率

比如我经常需要重启apache2，有时候命令行记不住，总是要去百度或者用上下键翻来翻去。那么可以编写脚本解决这个问题！

1）建一个文件夹存放脚本
root@ServerA:~# mkdir script
2）将该文件添加到PATH里（2和7选一个）
root@ServerA:~# PATH=$PATH:/root/script
3）新建文件
root@ServerA:~# cd script
root@ServerA:~/script# vim apR
4）将命令写入脚本并保存
1 #!/bin/bash
2 # This script can restart the apache2 server
3 sudo /etc/init.d/apache2 restart
5）更改权限
root@ServerA:~/script# chmod u+x apR

也可以更改这个文件夹的权限

root@ServerA:~/script# chmod -R 755 /root/script
//—R 递归更改权限
6）运行脚本
root@ServerA:~/script# ./apR
[ ok ] Restarting apache2 (via systemctl): apache2.service.

但是有个弊端，只能在script目录下运行脚本，那么需要解决这个问题

7）将脚本目录加入到环境变量
root@ServerA:~# vim $HOME/.bashrc

放在 /etc/profile随着系统更新会失效，6.6节有提到。 在里面加入：

export PATH="$PATH":/root/script/

保存，重启系统才能生效。或者 source

root@ServerA:~# reboot
root@ServerA:~# source /etc/profile
8）在任意界面运行脚本
root@ServerA:~# apR
[ ok ] Restarting apache2 (via systemctl): apache2.service.

还不用加./就能运行，很是方便。

注意如果脚本里写了cd，那么脚本运行的时候会创建一个子shell然后子shell里cd到了目录。运行的时候加个. 即可在当前shell生效。

Ubuntu 16.04 更新openssl至1.1.1

因为要用SM2国密算法，而Ubuntu自带的openssl不支持SM2。需要将openssl更新至1.1.1，网上的教程不齐就单独写一个。

1 查看版本

先看看自己的版本是多少：

openssl version

一般是很旧的，不会是最新的。

2 下载并解压源码

cd /usr/local/src
wget https://www.openssl.org/source/openssl-1.1.1-pre8.tar.gz
tar -zxf openssl-1.1.1-pre8.tar.gz

3 编译安装

进入自己的安装目录进行编译

cd openssl-1.1.1-pre8
./config
make install

4 移除旧版本 Openssl

这个看你自己的目录，我的是在/usr/bin里。找不到可以用find -name openssl找一下

mv /usr/bin/openssl /tmp/

再将编译好的软连接到源目录

ln -s /usr/local/bin/openssl /usr/bin/openssl

5 复制源码

如果出现：

openssl: error while loading shared libraries: libssl.so.1.1: cannot open shared object file: No such file or directory

那么将源码里的两个文件复制到系统目录：

cd /usr/local/src
cp libssl.so.1.1 /lib/x86_64-linux-gnu
libcrypto.so.1.1 /lib/x86_64-linux-gnu

这个时候就升级成功了！ 输入openssl version： 参考： https://w3ctech.com/topic/1873 https://sb.sb/blog/centos-upgrade-openssl/

Linux下core dump调试

linux``gdb``dump``posts

1 前言

很多时候在跑代码的时候会碰到以下错误： 然后有时候又不能快速定位错误的位置，这个时候就需要进行GDB的段错误调试。

通常情况下，core文件会包含了程序运行时的内存，寄存器状态，堆栈指针，内存管理信息还有各种函数调用堆栈信息等，我们可以理解为是程序工作当前状态存储生成第一个文件，许多的程序出错的时候都会产生一个core文件，通过工具分析这个文件，我们可以定位到程序异常退出的时候对应的堆栈调用等信息，找出问题所在并进行及时解决。

2 准备工作

2.1 core文件大小

需要设置当前会话的ulimit，若为0则不会产生对应的coredump。

命令：ulimit -a

命令：ulimit -c unlimited

这个时候如果产生了段错误，那么就会生成core文件。

2.2 core文件位置

第一次找了半天没找到core文件在哪，原来是自己没有设置好。

core文件默认的存储位置与对应的可执行程序在同一目录下，文件名是core，大家可以通过下面的命令看到core文件的存在位置：

命令：cat /proc/sys/kernel/core_pattern

可以看到当前产生的core文件保存在/var/core/文件目录下。也可以自行对其修改。

3 core调试

先写一个比较典型的内存越界错误的程序：

#include<stdio.h>

int main()
{
    int *a = NULL;
    printf("%d\n",*a);
    return 0;
}

注意

在编译的时候要加-g选项！一般release版不要加-g选项，这样编译时间会很久。

编译运行

可以看到有core dumped提示，那么这个时候可以用gdb进行调试。 但是请不要直接gdb core文件，会产生： not in executable format: File format not recognized 错误。

有些教程是直接让你gdb core文件。

需要这样： gdb 执行文件 core文件

gdb error /var/core/core-1528593396-error-20168-18446744073709551615

共享内存

发表于 2018-06-06 更新于 2020-04-14 分类于 Linux

共享内存

《Unix网络编程：卷2》``共享内存``posts

这个已经在之前的代码里使用过了，的确很方便，需要认真学习。

1 概述

共享内存区是IPC形式化中最快的。一旦这样的内存区映射到共享它的进程的地址空间，这些进程间数据的传递就不再涉及到内核。

使用共享内存可以大量减少系统开销。

要注意，fork之后的子进程并不与父进程共享内存区。子进程会拷贝父进程的内存区，然后单独使用。

2 映射函数

2.1 mmap

mmp函数把一个文件或一个Posix共享内存区对象映射到调用进程的地址空间。

#include <sys/mman.h>
void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);
  • addr指定描述符fd应被映射到的进程内空间的起始地址。通常为空，这样告诉内核自己去选择起始地址；
  • len是映射到调用进程地址空间中的字节数。
  • prot参数是读写权限，PORT_READ（数据可读）等等；
  • flags参数指定改动是否对进程可见，如：
    • MAP_SHARED，变动是共享的；
    • MAP_PRIVATE，变动是不共享的。

父子进程之间共享内存区的方法之一是，父进程在调用fork前先指定MAP_SHARED调用mmap。

2.2 mumap

从某个进程的地址空间删除一个映射关系。

int munmap(void *addr, size_t length);

addr是mmap返回的地址，len是映射区的大小。

2.3 msync

调用方msync实现硬盘的文件内容和内存映射区的内容一致。也就是说如果修改了内存映射到某个文件的内存区中内容，那么相应的内核会更新文件。

int msync(void *addr, size_t length, int flags);

flags参数：

  • MS_ASYNC，执行异步写，立即返回；
  • MS_SYNC，执行同步写，要等到写操作完成后才返回；
  • MS_INVALIDATE，最终副本不一致的文件数据的所有内存中副本都失效。
