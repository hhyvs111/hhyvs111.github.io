Tcpdump和Wireshark抓包简单教程

Posted on 2018-07-11 Edited on 2020-04-14 In Linux , 教程 , 计算机网络

Tcpdump和Wireshark抓包简单教程

总结一下抓包的过程，主要是一个熟悉流程的过程。

1 Linux抓包

抓包的一个传统流程，当然使用的是tcpdump。

1.1 定位

首先抓包不能一通乱抓，要确定你想知道哪个进程的网络数据交换过程，那么当然要知道进程ID和该进程所使用的端口。

主要是针对自己工作的SSLVPN抓包过程。

ps命令

ps -aux | grep 进程名

使用该命令获取进程的ID。

netstat命令

netstat -anp |grep 进程名

这也可以获得端口号和进程ID，直接省略了PS过程。

1.2 抓包

使用经典的tcpdump来抓包，建议保存到本地用wireshark看。

tcpdump -i eth0 port 8801 -c 100 -X -s 0 -w sgip.cap
//抓所有8801端口的数据包，其他的IP可以看网上教程

几个重要的选项：

  • -c，抓包的个数；
  • -X，显示包的内容；
  • -s 0，不截断，显示完整长度。

1.3 分析包（TCP包）

还有很多包，UDP、DNS等等。

主要是一些三次握手是否正常，断开是否异常等。 然后还可以看包体的数据： 这是前面4个是TCP/IP的4层模型，最后一个是包自己的协议数据。

突然想起以前自己做项目的时候，遇到网络问题没什么没有想起抓包呢？可能那个时候还不会吧，现在想来一抓包，问题都能分析出来了。

2 Window抓包

与Linux不同的是，Windows没有那么多便捷的命令可以直接查看数据。

2.1 定位

如果你想捕获某个程序的包，那么也需要知道其用的端口。可以在任务管理其查看。

  • 查看-选择列-勾选进程PID

  • 找到指定进程的PID

netstat命令

和Linux有些许不同。

netstat -ano | findstr 进程ID

这个时候就可以监控进程所使用的端口了。

2.2 抓包

这个时候可以直接使用wireshark进行抓包，只要在过滤条件下设置一下你要的端口或者IP就能显示你想要的数据了。

添加过滤条件显示

可以看一下这篇文章: wireshark过滤语法总结

分析过程就不再赘述了，可以参考这篇文章：WireShark抓包 图解探索网络请求过程（五层网络模型、三次握手、滑动窗口协议）

阿里云服务器Ubuntu下Apache2配置https

阿里云服务器Ubuntu下Apache2配置https

教程``posts``SSL

最近把网站搬迁了，然后HTTPS也得弄一弄，本文主要是介绍一下阿里云服务器如何给自己的网站配置https

1 购买证书

这个证书的下载比较鸡贼，阿里云把免费的给藏起来了，先点击这里： 然后再点击增强型，这个时候就会出现免费的证书选项了。 然后点击免费的购买

2 下载证书

将证书里的四个文件放在/etc/apache2/cert，自己建一个cert文件夹。

3 配置SSL

进入目录

cd /etc/apache2/sites-available

编辑default-ssl.conf文件，修改域名和目录

ServerAdmin webmaster@localhost
ServerName 你的域名:443
DocumentRoot /var/www/html/你的网站目录

然后在SSLEngine on的下面加入如下内容：

SSLProtocol all -SSLv2 -SSLv3
SSLCipherSuite HIGH:!RC4:!MD5:!aNULL:!eNULL:!NULL:!DH:!EDH:!EXP:+MEDIUM
SSLHonorCipherOrder on
SSLCertificateFile  /etc/apache2/cert/public.pem
SSLCertificateKeyFile /etc/apache2/cert/你的.key
SSLCertificateChainFile /etc/apache2/cert/chain.pem

4 配置HTTPS

编辑000-default.conf，在<VirtualHost *:80></VirtualHost>标签内随便一个地方加入以下三行:

RewriteEngine on
RewriteCond  %{HTTPS} !=on
RewriteRule  ^(.*) https://%{SERVER_NAME}$1 [L,R]

5 重启

记得载入模块：

a2ensite default-ssl
#然后重启apache2
sudo etc/init.d/apache2 restart

6 备注

记得目录一定要对，然后看一下阿里云服务器是否开放了443这个端口，配置一下安全组规则。 推荐一个wordpress插件：Really Simple SSL 但是如果有时候你关闭了SSL，要访问http形式的网站，但这个时候你进不了网站，关不了这个插件，可以参考这个强制关闭该插件的方法：

进入wordpress目录

cd wp-content/plugins/really-simple-ssl

修改文件

将改txt后缀改为php

mv force-deactivate.txt force-deactivate.php

用http访问该PHP

http://www.yourdomain.com/wp-content/plugins/really-simple-ssl/force-deactivate.php

这个时候就可以卸载该插件关闭全站https了。

<VirtualHost *:80>

    #这个不是根目录，只是说可以重写罢了

        Options FollowSymLinks

<VirtualHost *:80>
    ServerAdmin webmaster@localhost
    ServerName 我的域名
    DocumentRoot /var/www/html/tanwenbo

        Options FollowSymLinks
        AllowOverride All
    </Directory>
        CustomLog ${APACHE_LOG_DIR}/access.log combined
</VirtualHost>

这个时候访问我的域名可以直达我的网站，然后访问服务器IP可以直达导师的服务器。

Linux下编写小脚本提高效率

Posted on 2018-07-04 Edited on 2020-04-14 In Linux

Linux下编写小脚本提高效率

比如我经常需要重启apache2，有时候命令行记不住，总是要去百度或者用上下键翻来翻去。那么可以编写脚本解决这个问题！

1）建一个文件夹存放脚本
root@ServerA:~# mkdir script
2）将该文件添加到PATH里（2和7选一个）
root@ServerA:~# PATH=$PATH:/root/script
3）新建文件
root@ServerA:~# cd script
root@ServerA:~/script# vim apR
4）将命令写入脚本并保存
1 #!/bin/bash
2 # This script can restart the apache2 server
3 sudo /etc/init.d/apache2 restart
5）更改权限
root@ServerA:~/script# chmod u+x apR

也可以更改这个文件夹的权限

root@ServerA:~/script# chmod -R 755 /root/script
//—R 递归更改权限
6）运行脚本
root@ServerA:~/script# ./apR
[ ok ] Restarting apache2 (via systemctl): apache2.service.

但是有个弊端，只能在script目录下运行脚本，那么需要解决这个问题

7）将脚本目录加入到环境变量
root@ServerA:~# vim $HOME/.bashrc

放在 /etc/profile随着系统更新会失效，6.6节有提到。 在里面加入：

export PATH="$PATH":/root/script/

保存，重启系统才能生效。或者 source

root@ServerA:~# reboot
root@ServerA:~# source /etc/profile
8）在任意界面运行脚本
root@ServerA:~# apR
[ ok ] Restarting apache2 (via systemctl): apache2.service.

还不用加./就能运行，很是方便。

注意如果脚本里写了cd，那么脚本运行的时候会创建一个子shell然后子shell里cd到了目录。运行的时候加个. 即可在当前shell生效。

Ubuntu 16.04 更新openssl至1.1.1

Posted on 2018-06-28 Edited on 2020-04-14 In Linux , 教程

Ubuntu 16.04 更新openssl至1.1.1

因为要用SM2国密算法，而Ubuntu自带的openssl不支持SM2。需要将openssl更新至1.1.1，网上的教程不齐就单独写一个。

1 查看版本

先看看自己的版本是多少：

openssl version

一般是很旧的，不会是最新的。

2 下载并解压源码

cd /usr/local/src
wget https://www.openssl.org/source/openssl-1.1.1-pre8.tar.gz
tar -zxf openssl-1.1.1-pre8.tar.gz

3 编译安装

进入自己的安装目录进行编译

cd openssl-1.1.1-pre8
./config
make install

4 移除旧版本 Openssl

这个看你自己的目录，我的是在/usr/bin里。找不到可以用find -name openssl找一下

mv /usr/bin/openssl /tmp/

再将编译好的软连接到源目录

ln -s /usr/local/bin/openssl /usr/bin/openssl

5 复制源码

如果出现：

openssl: error while loading shared libraries: libssl.so.1.1: cannot open shared object file: No such file or directory

那么将源码里的两个文件复制到系统目录：

cd /usr/local/src
cp libssl.so.1.1 /lib/x86_64-linux-gnu
libcrypto.so.1.1 /lib/x86_64-linux-gnu

这个时候就升级成功了！ 输入openssl version： 参考： https://w3ctech.com/topic/1873 https://sb.sb/blog/centos-upgrade-openssl/

Linux下core dump调试

linux``gdb``dump``posts

1 前言

很多时候在跑代码的时候会碰到以下错误： 然后有时候又不能快速定位错误的位置，这个时候就需要进行GDB的段错误调试。

通常情况下，core文件会包含了程序运行时的内存，寄存器状态，堆栈指针，内存管理信息还有各种函数调用堆栈信息等，我们可以理解为是程序工作当前状态存储生成第一个文件，许多的程序出错的时候都会产生一个core文件，通过工具分析这个文件，我们可以定位到程序异常退出的时候对应的堆栈调用等信息，找出问题所在并进行及时解决。

2 准备工作

2.1 core文件大小

需要设置当前会话的ulimit，若为0则不会产生对应的coredump。

命令：ulimit -a

命令：ulimit -c unlimited

这个时候如果产生了段错误，那么就会生成core文件。

2.2 core文件位置

第一次找了半天没找到core文件在哪，原来是自己没有设置好。

core文件默认的存储位置与对应的可执行程序在同一目录下，文件名是core，大家可以通过下面的命令看到core文件的存在位置：

命令：cat /proc/sys/kernel/core_pattern

可以看到当前产生的core文件保存在/var/core/文件目录下。也可以自行对其修改。

3 core调试

先写一个比较典型的内存越界错误的程序：

#include<stdio.h>

int main()
{
    int *a = NULL;
    printf("%d\n",*a);
    return 0;
}

注意

在编译的时候要加-g选项！一般release版不要加-g选项，这样编译时间会很久。

编译运行

可以看到有core dumped提示，那么这个时候可以用gdb进行调试。 但是请不要直接gdb core文件，会产生： not in executable format: File format not recognized 错误。

有些教程是直接让你gdb core文件。

需要这样： gdb 执行文件 core文件

gdb error /var/core/core-1528593396-error-20168-18446744073709551615

共享内存

Posted on 2018-06-06 Edited on 2020-04-14 In Linux

共享内存

《Unix网络编程：卷2》``共享内存``posts

这个已经在之前的代码里使用过了，的确很方便，需要认真学习。

1 概述

共享内存区是IPC形式化中最快的。一旦这样的内存区映射到共享它的进程的地址空间，这些进程间数据的传递就不再涉及到内核。

使用共享内存可以大量减少系统开销。

要注意，fork之后的子进程并不与父进程共享内存区。子进程会拷贝父进程的内存区，然后单独使用。

2 映射函数

2.1 mmap

mmp函数把一个文件或一个Posix共享内存区对象映射到调用进程的地址空间。

#include <sys/mman.h>
void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);
  • addr指定描述符fd应被映射到的进程内空间的起始地址。通常为空，这样告诉内核自己去选择起始地址；
  • len是映射到调用进程地址空间中的字节数。
  • prot参数是读写权限，PORT_READ（数据可读）等等；
  • flags参数指定改动是否对进程可见，如：
    • MAP_SHARED，变动是共享的；
    • MAP_PRIVATE，变动是不共享的。

父子进程之间共享内存区的方法之一是，父进程在调用fork前先指定MAP_SHARED调用mmap。

2.2 mumap

从某个进程的地址空间删除一个映射关系。

int munmap(void *addr, size_t length);

addr是mmap返回的地址，len是映射区的大小。

2.3 msync

调用方msync实现硬盘的文件内容和内存映射区的内容一致。也就是说如果修改了内存映射到某个文件的内存区中内容，那么相应的内核会更新文件。

int msync(void *addr, size_t length, int flags);

flags参数：

  • MS_ASYNC，执行异步写，立即返回；
  • MS_SYNC，执行同步写，要等到写操作完成后才返回；
  • MS_INVALIDATE，最终副本不一致的文件数据的所有内存中副本都失效。

System V信号量

Posted on 2018-06-01 Edited on 2020-04-14 In Linux , 计算机网络

System V信号量

《Unix网络编程：卷2》``信号量``posts

System V信号量和Posix信号量表面上区分是从函数命名方式上，前者没有下划线。后面再总结具体区分。

1 概述

System V信号量比Posix信号量增加了另外一级复杂度：

  • 计数信号量集：一个或多个信号量（构成一个集合），其中每个都是计数信号量。一般讨论System V的是计数信号量集，而Posix是指单个计数信号量。

semid_ds 信号量集结构如下（Linux下没有sem参数，但其实是有的）：

struct semid_ds {
       struct ipc_perm sem_perm;  /* Ownership and permissions */
       time_t          sem_otime; /* Last semop time */
       time_t          sem_ctime; /* Last change time */
       unsigned long   sem_nsems; /* No. of semaphores in set */
   };

ipc_perm之前有提到是访问权限。

man semop结果：

Each semaphore in a System V semaphore set has the following associated values:

struct sem{
       unsigned short  semval;   /* semaphore value */
       unsigned short  semzcnt;  /* # waiting for zero */
       unsigned short  semncnt;  /* # waiting for increase */
       pid_t           sempid;   /* ID of process that did last op */
    };

2 相关函数

2.1 semget

#include <sys/types.h>
#include <sys/ipc.h>
#include <sys/sem.h>

int semget(key_t key, int nsems, int semflg);

返回值是一个称为信号量标识符的整数，后面的函数会用到。

  • nsems是指定集合中的信号量数。
  • semflg就是之前的oflag，权限位。

创建新信号集的时候，相应的semid_ds结构体有些成员会被初始化：

sem_perm.cuid  and  sem_perm.uid  are  set to the effective user ID of thecalling process.

sem_perm.cgid and sem_perm.gid are set to the effective group  ID  of  thecalling process.

The least significant 9 bits of sem_perm.mode are set to the least significant 9 bits of semflg.

sem_nsems is set to the value of nsems.

sem_otime is set to 0.

sem_ctime is set to the current time.

2.2 semop

int semop(int semid, struct sembuf *sops, size_t nsops);

int semtimedop(int semid, struct sembuf *sops, size_t nsops, const struct timespec *timeout);
  • nops表明sembuf结构体的个数，因为是指针形式保存的结构体，所以需要指明个数；
  • sops指向结构体数组首地址。

主要看一下sembuf结构体：

struct sembuf{
       unsigned short sem_num;  /* semaphore number */
       short          sem_op;   /* semaphore operation */
       short          sem_flg;  /* operation flags */
}

与单个信号量相关的几个值： semval：信号量的当前值，在文章开头信号量的结构中已提到。 semncnt：等待semval变为大于当前值的线程数。在文章开头信号量的结构中已提到。 semzcnt：等待semval变为0的线程数。在文章开头信号量的结构中已提到。 semadj ：指定信号量针对某个特定进程的调整值。只有sembuf结构的sem_flag指定为SEM_UNDO后，semadj才会随着sem_op而更新。讲简单一点：对某个进程，在指定SEM_UNDO后，对信号量semval值的修改都会反应到semadj上，当该进程终止的时候，内核会根据semadj的值，重新恢复信号量之前的值。

  • sem_num为信号量数组下标，直到nsems-1（semget指定的参数）；
  • sem_op有三类值：
    • sem_op > 0，将值加到semval上（sem结构体）；
    • sem_op < 0，有多种情况：
      • 如果semval大于或等于sem_op的值，那么semval - sem_op；
      • 如果semval小于sem_op的值，相应的信号量semncnt值加1，调用线程将阻塞，直到semval大于或等于sem_op的值。到时候线程将被解除阻塞，然后还要semval - sem_op，semncnt 减1。
    • sem_op = 0，也有多种情况：
      • semval为0返回；
      • semval不为0，相应信号量的semzcnt加1，阻塞线程直到semval为0（如果指定了IPC_NOWAIT则调用线程不会被投入睡眠）

（详细解析参考书籍230页）

  • sem_flag，信号量操作的属性标志：
    • 如果为0，表示正常操作；
    • 如果为IPC_WAIT，使对信号量的操作时非阻塞的。即指定了该标志，调用线程在信号量的值不满足条件的情况下不会被阻塞，而是直接返回-1，并将errno设置为EAGAIN；
    • 如果为SEM_UNDO，那么将维护进程对信号量的调整值，以便进程结束时恢复信号量的状态。

2.3 semctl

int semctl(int semid, int semnum, int cmd, ... /* union semun arg */);

对一个信号量执行各种控制操作。

  • semnum为信号集内数组下标；

  • cmd值参考书籍231页。

  • semun，可选联合体，定义如下：

    union semun {

    int              val;    /* Value for SETVAL */
struct semid_ds *buf;    /* Buffer for IPC_STAT, IPC_SET */
unsigned short  *array;  /* Array for GETALL, SETALL */
struct seminfo  *__buf;  /* Buffer for IPC_INFO (Linux-specific) */

    };

seminfo具体结构在man semctl中有。Linux有在头文件定义该值。

3 区别

主要总结一下Posix和System V信号量的区别（来自网上）。

  • XSI system V的信号量是信号量集，可以包括多个信号灯（有个数组），每个操作可以同时操作多个信号灯； posix是单个信号灯，POSIX有名信号灯支持进程间通信，无名信号灯放在共享内存中时可以用于进程间通信。
  • POSIX信号量在有些平台并没有被实现，比如：SUSE8，而SYSTEM V大多数LINUX/UNIX都已经实现。两者都可以用于进程和线程间通信。但一般来说，system v信号量用于 进程间同步、有名信号灯既可用于线程间的同步，又可以用于进程间的同步、posix无名用于同一个进程的不同线程间，如果无名信号量要用于进程间同步，信号量要放在共享内存中。
  • POSIX有两种类型的信号量，有名信号量和无名信号量。有名信号量像system v信号量一样由一个名字标识。
  • POSIX通过sem_open单一的调用就完成了信号量的创建、初始化和权限的设置，而system v要两步。也就是说posix 信号是多线程，多进程安全的，而system v不是，可能会出现问题。
  • system V信号量通过一个int类型的值来标识自己（类似于调用open()返回的fd），而sem_open函数返回sem_t类型（长整形）作为posix信号量的标识值。
  • 对于System V信号量你可以控制每次自增或是自减的信号量计数，而在Posix里面，信号量计数每次只能自增或是自减1。
  • Posix无名信号量提供一种非常驻的信号量机制。

读写锁

Posted on 2018-05-31 Edited on 2020-04-14 In Linux

读写锁

《Unix网络编程：卷2》``读写锁``posts

新的概念，和之前的互斥锁有点不同。

1 概述

读写锁，顾名思义。对读和写加锁，但其实不然。与互斥锁类似，不过读写锁有个特点，那就是写独占，读共享。

规则如下：

  • 只要没有线程持有某个给定的读写锁用于写，那么任意数目的线程可以持有该读写锁用于读,也就是“写独占，读共享”；
  • 仅当没有线程持有某个给定的读写锁用于读或用于写时，才能分配该读写锁用于写。

就是没有线程在修改数据，这个时候大家都可以读。只要没人在读或者在修改，那么线程才可以去修改它。

个人认为读写锁在数据库的增删查改的时候很有必要！

2 获取与释放读写锁

2.1 读出锁

#include <pthread.h>

int pthread_rwlock_rdlock(pthread_rwlock_t *rwlock);

如果对应的读写锁已有某个写入者持有，那么会阻塞调用线程。

2.2 写入锁

int pthread_rwlock_wrlock(pthread_rwlock_t *rwlock);

如果对应的读写锁已由另一个写入者持有，或者已由一个或多个读写者持有，那就阻塞调用线程。

写的时候不能有人在读。

2.3 释放锁

int pthread_rwlock_unlock(pthread_rwlock_t *rwptr);

2.4 尝试访问

如果不能马上取得，那么返回EBUSY错误。

int pthread_rwlock_tryrdlock(pthread_rwlock_t *rwlock);

int pthread_rwlock_trywrlock(pthread_rwlock_t *rwlock);

以上返回值都一样：成功返回0，失败返回错误号。
